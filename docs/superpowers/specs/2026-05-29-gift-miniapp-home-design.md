# 送礼小程序首页与导航设计方案

## 状态

已确认：第一期首页采用极简启动页，不在首页直接展示问答选项、商品、社区内容流。

## 第一阶段范围

第一期只解决一个核心动作：让用户进入送礼推荐流程。

不在第一期首页承载：

- 社区内容流
- 礼物偏好广场
- 商品推荐列表
- 送礼对象和场景的直接大面积选择
- 复杂解释文案

这些能力后续可以扩展，但不挤占第一期首页。

## 首页结构

首页采用极简结构：

1. 顶部保留产品名：`送了么`
2. 页面中部只放一个主按钮：`开始选礼物`
3. 底部放两个 Tab：`首页`、`我的`

首页不放以下文案：

- `不知道送什么？`
- `回答几个问题，帮你挑一份有分寸的礼物。`
- `恋人 / 父母 / 生日 / 纪念日`

原因：这些内容会让首页变成半个问答页。第一期首页应该更安静，把决策过程放到点击按钮之后。

## 交互路径

用户进入小程序后：

1. 看到极简首页
2. 点击 `开始选礼物`
3. 进入一步步问答流程
4. 完成问答后得到礼物推荐结果

首页只负责启动，不负责收集信息。

## 导航设计

第一期使用底部 Tab，不使用左上角三横菜单。

底部 Tab：

- `首页`：启动送礼推荐流程
- `我的`：管理用户相关内容

选择底部 Tab 的原因：

- 符合微信小程序常见使用习惯
- `我的` 入口清晰，不会被隐藏
- 后续扩展社区时，可以自然升级为 `首页 / 社区 / 我的`
- 避免左上角三横菜单和微信小程序顶部胶囊按钮区域产生拥挤感

## 我的页第一期职责

`我的` 页第一期只做轻量管理，不做复杂社区。

建议包含：

- 我的偏好
- 收礼人档案
- 历史推荐

其中 `收礼人档案` 用来维护常送礼对象，例如恋人、父母。后续如果产品需要更强的个性化推荐，可以从这里扩展。

## 后续扩展预留

后续如果加入社区能力，首页不需要推翻，只需要扩展导航和内容层级：

- 第一阶段：`首页 / 我的`
- 第二阶段：`首页 / 社区 / 我的`

社区可以承载：

- 用户发布自己的礼物偏好
- 用户分享送礼经验
- 节日礼物灵感
- 真实收礼反馈

但社区不作为第一期首页核心内容。

## 推荐执行方案

第一期推荐不默认接入在线大模型。主要原因是成本、速度和稳定性。

推荐链路拆成离线和在线两部分：

```text
离线：攻略/经验/运营整理 -> AI 辅助抽取结构化标签和推荐理由 -> 审核 -> 入库
在线：用户问答结果 -> 轻量决策模型/决策表 -> 返回推荐候选
```

在线阶段不依赖大模型实时生成核心推荐。大模型可以作为离线建库工具，或后续用于生成更自然的表达文案，但不作为第一期推荐决策核心。

### 第一阶段：云函数 + 决策表/轻量树模型

第一期推荐服务部署在 CloudBase 云函数中。

云函数只负责在线推理：

```text
用户答案 -> 特征转换 -> 决策表/轻量树模型 -> 推荐候选
```

适合放在云函数里的模型形态：

- 可配置决策表
- 轻量决策树
- 小型 GBDT 的导出结果
- JSON 形式的树结构
- 纯 JavaScript 或纯 Python 推理代码

不建议在云函数里做：

- 模型训练
- 大型依赖安装
- 复杂 native 库推理
- 依赖 `sklearn`、`lightgbm`、`catboost` 运行时的重型在线服务

更稳的做法是离线训练或配置模型，然后导出为可直接执行的轻量产物，例如 `model.json`。云函数运行时只加载模型文件并遍历树结构。

### 第二阶段：云函数 + 导出的 GBDT 模型

当积累到用户反馈数据后，可以升级为更强的 CPU 排序模型。

可用反馈包括：

- 点击
- 收藏
- 换一批
- 不喜欢
- 最终购买或跳转
- 用户主动反馈“合适/不合适”

训练仍然放在线下完成。训练完成后，将模型导出成轻量推理格式，再部署到云函数。

### 第三阶段：云托管承载复杂模型

如果后续模型变重，例如需要直接使用 Python 的 LightGBM、CatBoost、sklearn，或希望模型常驻内存降低延迟，则迁移到 CloudBase 云托管。

云托管适合：

- 复杂 Python 依赖
- native 库
- 更大的模型文件
- 更稳定的低延迟服务
- 可配置 CPU/内存和最小实例数

代价是服务形态更重，运行成本也更持续。

### 当前结论

第一期采用：

```text
离线建库 + CloudBase 云函数 + 决策表/轻量树模型
```

训练和建库离线完成，线上只做快速、稳定、可解释的推荐推理。

## 离线建库结构化字段

离线建库时，AI 不直接生成线上推荐，而是辅助把攻略、经验和运营材料抽取成结构化礼物知识。

抽取目标不是保存原文，而是沉淀可被轻量决策模型使用的字段。

### 字段分层

标签和字段分为三类：

```text
核心受控字段：强枚举，直接参与过滤和决策
半受控标签：有主标签池，可审核扩充，用于匹配和排序
自定义标签：保留原词，只做辅助召回和文案，不直接主导推荐
```

### 核心受控字段

核心字段必须提前定义枚举，避免标签泛滥。

建议包含：

- 送礼对象
- 场景
- 亲密度/关系阶段
- 预算区间
- 礼物大类
- 准备时效
- 交付方式
- 风险标签

这些字段可以直接影响过滤。例如用户选择“今天就要送”，系统应过滤掉“需要提前定制”“7 天以上”“需要手作/准备”的礼物方向。

### 亲密度/关系阶段

第一期主打恋人和父母，但两类对象的关系字段不完全相同。

恋人方向：

- 暧昧期
- 刚在一起
- 稳定情侣
- 长久情侣
- 异地恋
- 即将见家长/进入更严肃阶段

父母方向更适合用表达类型，而不是亲密度：

- 日常关心
- 生日重视
- 节日问候
- 补偿陪伴
- 改善生活质量
- 健康关怀

### 兴趣标签

兴趣标签表示“收礼人喜欢什么”，不等同于礼物风格。

建议拆成两层：

```text
interest_category：兴趣大类
interest_tags：兴趣细标签
```

示例：

```json
{
  "interest_category": ["娱乐追星", "音乐"],
  "interest_tags": ["K-pop", "演唱会", "周边收藏"],
  "custom_tags": ["BLACKPINK"]
}
```

AI 抽取时优先映射到已有标签。如果没有合适标签，再生成候选新标签，进入审核，不直接上线。

### 视觉风格标签

视觉风格标签表示“礼物看起来或表达出来是什么气质”，和兴趣分开。

建议标签池：

- 高级感
- 可爱
- 极简
- 复古
- 文艺
- 科技感
- 温柔
- 精致
- 自然
- 潮流
- 国风
- 低调
- 隆重

`文艺` 更适合作为视觉风格；兴趣里应尽量使用更具体的词，例如阅读、展览、摄影、手作、音乐。

### 实用性标签

实用性是礼物属性，不归入兴趣标签。

建议标签：

- 高实用
- 中实用
- 低实用
- 日常消耗
- 长期使用
- 改善生活质量
- 装饰纪念
- 情绪价值为主

示例：

- 按摩仪：高实用、长期使用、改善生活质量
- 鲜花：低实用、情绪价值为主、装饰纪念
- 咖啡礼盒：中实用、日常消耗
- 定制相册：低实用、装饰纪念、情绪价值为主

### 时效性标签

时效性单独建字段，不能混入兴趣或风格。

建议拆成两个字段：

```text
preparation_time：准备/到手时间
delivery_mode：交付方式
```

准备/到手时间：

- 当天可送
- 1-3 天
- 3-7 天
- 7 天以上
- 需要提前定制

交付方式：

- 即时数字礼物
- 同城可达
- 快递可达
- 线下购买
- 需要预约
- 需要手作/准备

### 情绪价值标签

情绪价值表示这份礼物想表达什么。

建议标签：

- 浪漫
- 陪伴
- 感谢
- 照顾
- 惊喜
- 纪念
- 尊重
- 体面
- 治愈
- 安全感

### 搭配关系

礼物方向应记录适合搭配什么，方便后续生成组合推荐。

轻量字段：

```text
pairing_tags：手写卡片、鲜花、蛋糕、包装、照片、晚餐、红包、小饰品
```

如果需要更强表达，可以扩展为结构化数组：

```json
{
  "pairing_suggestions": [
    {
      "with": "手写卡片",
      "reason": "补足情绪表达，让礼物不显得只靠形式"
    },
    {
      "with": "甜品/蛋糕",
      "reason": "适合生日或纪念日当天制造仪式感"
    }
  ]
}
```

第一期可以先用 `pairing_tags`，后续再扩展为带理由的结构。

### 风险标签

风险标签用于避免推荐翻车，优先级高于普通匹配标签。

建议标签：

- 过度亲密
- 太私人
- 审美依赖强
- 尺码风险
- 香味风险
- 肤质风险
- 价格敏感
- 长辈不易理解
- 容易显得敷衍
- 不适合暧昧期
- 不适合父母

### 礼物方向记录示例

```json
{
  "gift_direction": "定制相册/照片书",
  "target": ["恋人"],
  "scene": ["纪念日", "生日"],
  "intimacy_stage": ["稳定情侣", "长久情侣", "异地恋"],
  "budget_range": ["100-300", "300-800"],
  "interest_category": ["生活记录", "摄影"],
  "interest_tags": ["照片", "回忆整理", "手作感"],
  "visual_style_tags": ["文艺", "温柔", "低调"],
  "utility_tags": ["低实用", "装饰纪念", "情绪价值为主"],
  "preparation_time": ["3-7 天", "需要提前定制"],
  "delivery_mode": ["快递可达", "需要手作/准备"],
  "emotional_tags": ["陪伴", "纪念", "安全感"],
  "pairing_tags": ["手写卡片", "鲜花", "晚餐"],
  "risk_tags": ["不适合暧昧期", "需要共同回忆素材"],
  "custom_tags": [],
  "recommend_reason": "适合已经有共同经历的关系，把回忆实体化，表达稳定陪伴。",
  "boundary_note": "关系越稳定越适合；关系还没确定时容易显得过重。",
  "source_type": "攻略抽取"
}
```

### 在线使用原则

字段在线使用分工：

```text
硬过滤：target、scene、intimacy_stage、budget_range、preparation_time、delivery_mode、risk_tags
匹配排序：interest_category、interest_tags、visual_style_tags、utility_tags、emotional_tags、pairing_tags
辅助表达：custom_tags、recommend_reason、boundary_note
```

## 结果页展示方案

结果页采用“方案优先，但不报告化”的结构。

核心原则：

- 先给一句自然语言判断
- 不展示顶层关系标签
- 不展示机械评分
- 不直接把用户丢进商品列表
- 推荐品用小元素展示，一屏尽量看到更多候选
- 每个推荐品只展示 1-2 个短亮点

### 顶部说明

结果页顶部不放大标题，例如不使用：

```text
送一份有纪念感的小礼物
```

改为一句短判断：

```text
这次更适合选有回忆感、低压力、能马上说清心意的礼物。
```

可以补一条更克制的说明：

```text
已避开过度贵重和太私人化的选择
```

这类说明只表达推荐边界，不暴露用户具体选择的关系状态。

### 不展示顶层关系标签

结果页不展示如下标签：

- 稳定情侣
- 暧昧期
- 刚在一起
- 吵架冷战
- 能对着放屁

这些信息只用于后台推荐判断，不直接暴露给用户。

原因：

- 关系状态属于用户隐私和产品判断
- 直接展示会显得产品在给用户关系定性
- 商品本身已有亮点标签，顶层关系标签会重复且冒犯感更强

### 推荐品展示

推荐品不使用表格，使用一个个可点击的小元素。

一屏至少展示 6 个候选，后续可以横滑查看更多。

每个推荐品结构：

```text
图片/占位图
名称
1-2 个短亮点
价格或风险提示
```

短亮点示例：

- 有回忆
- 能保存
- 当天送
- 仪式感
- 氛围好
- 放松
- 不贵重
- 真诚
- 好出片
- 精致

风险也可以用短词提示：

- 香味风险
- 看审美
- 需提前
- 尺码风险

### 推荐搭配

结果页保留推荐搭配区域，但保持简短。

示例：

```text
照片书 + 卡片：回忆感更强
鲜花 + 甜品：当天更稳
```

搭配建议的作用是让用户知道如何把单个礼物变成一份完整方案。

### 底部操作

结果页底部固定两个操作：

```text
换方案 / 看更多
```

含义：

- `换方案`：当前方向不满意，重新生成另一组推荐
- `看更多`：进入更多推荐品或商品详情

### 当前结果页定稿

第一期结果页结构：

```text
一句短判断
推荐品小元素区
推荐搭配
底部：换方案 / 看更多
```

关系阶段、亲密度、冲突状态等只进入推荐逻辑，不直接展示在结果页。

## 当前定稿

第一期首页定稿方向：

```text
顶部：送了么
中间：开始选礼物
底部：首页 / 我的
```

点击 `开始选礼物` 后，再进入送礼问答流程。
